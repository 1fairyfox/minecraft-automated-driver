### CI findings — the real first-run flow of a spawned production client (worth keeping)

The spawn smoke failed three times in CI before passing — each a real fact about a
freshly-spawned production client, NOT a driver bug (spawn/handshake/connect succeeded
every time). Recorded so we don't relearn them:

1. **The agent handshake is written during init, before the "Loading Minecraft" splash
   (`class_424`) clears.** So `client_spawn` returning ≠ the UI is drivable yet; a driver
   must poll the semantic screen for readiness, not assume.
2. **A production client runs on intermediary mappings**, so screen *class* names are
   `class_XXX`, not yarn (`TitleScreen`). Key on widget *labels*, never class names.
3. **A fresh run dir opens the "Welcome to Minecraft!" accessibility onboarding
   (`class_8032`) before the title screen** — a real driver clicks "Continue" through it.
4. **The title-screen Options button's label is literally `"Options..."` (trailing
   ellipsis).** Exact-match readiness on "Options" never fires; detect the title screen on
   `"Singleplayer"` (stable, ellipsis-free). The agent's click-by-name still resolves
   "Options" → "Options..." (the gametest proves it).

The smoke now drives the real path: splash → onboarding (Continue) → title (Singleplayer)
→ click Options by name → refuse bad token → kill.
